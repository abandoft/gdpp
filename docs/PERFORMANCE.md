# GDPP AOT 体积与性能基线

本文记录 2026-07-14 在同一台 Windows 测试机上对 Dungeon Rush 原始 GDScript Release
与 GDPP AOT Release 的对照结果。数据用于识别真实收益和回归，不把单个项目的结果外推为
“所有游戏都更快”或“AOT 必然减小包体”。原始日志保存在本地
`build/windows-qa/comparison/`，该目录是可删除、不可交付的测试证据目录。

## 测试环境

| 项目 | 配置 |
|---|---|
| 操作系统 | Windows 11 x86_64，Build 26200 |
| CPU | AMD Ryzen 7 5800U，8 核 16 线程 |
| GPU | AMD Radeon Graphics，驱动 31.0.21921.1000 |
| 内存 | 约 13.9 GiB 可见内存 |
| Godot | 官方 4.5.2 stable，标准精度，Release 导出 |
| 项目 | Dungeon Rush，197 个脚本编译单元 |
| 对照 | `ori` 保留 GDScript 字节码；`aot` 剥离 `.gd/.gdc` 并加载项目 Release 原生库 |

启动测试交替运行两个版本各 8 次。游戏测试使用相同自动化输入进入 Home 场景，分别执行
3 次垂直同步和 3 次解除帧率上限的 20 秒采样。启动时间采用中位数，运行指标采用 3 次均值。

## 发行体积

| 指标 | ori | aot | AOT 变化 |
|---|---:|---:|---:|
| PCK | 63,949,740 B | 64,131,952 B | +182,212 B（+0.285%） |
| 项目运行库/回退库 | 101,376 B | 4,206,592 B | +4,105,216 B |
| 完整导出目录 | 164,697,156 B | 168,984,584 B | +4,287,428 B（+2.603%） |
| PCK 内 `.gd/.gdc` | 195 个 `.gdc`，263,661 B | 0 | -263,661 B |

下表是首轮诊断导出的未压缩内容拆分；后续默认值、导出缓存和生命周期修复使压缩后 AOT PCK
再减少 3,424 B，但没有改变主要结论。AOT PCK 变大的直接原因是场景转换，不是源码或中间产物
泄漏：

| PCK 未压缩内容变化 | 字节 |
|---|---:|
| 删除 GDScript 字节码 | -263,661 |
| 317 个 `.scn` 转为原生脚本类后的序列化增长 | +488,668 |
| `.remap` 减少 | -9,519 |
| 生成的 3 个 Autoload 场景 | +336 |
| 其他项目元数据净变化 | +147 |
| 未压缩净变化 | +215,971 |
| 压缩后 PCK 净变化 | +185,636 |

根因是原生成构造函数在编辑器中跳过全部字段初始化，使 ClassDB 收集到的原生默认值为空，
PackedScene 因而显式写入更多属性。当前构造函数会在编辑器中执行可证明无副作用的字面量、集合
和 Godot 值类型默认值，同时继续延迟单例、资源加载和用户调用；该修复只消除了其中一小部分
序列化增长，剩余差异仍需逐场景归因。当前 AOT 包实际有 1953 个项目文件；独立审计加载了
331 个场景/资源，并确认 `.gd/.gdc`、compiler、SDK、生成 C++、对象文件和 fallback 泄漏数
均为 0，完整目录恰好包含一个 GDPP 项目动态库。

结论是：源码保护已经成立，但当前项目的 AOT PCK 并没有缩小。后续应在不改变场景语义的前提下
恢复默认值/覆盖值判定，给场景序列化增长设置回归预算；在完成前不能宣传“AOT 自动缩包”。

## 导出耗时

| 场景 | ori | aot | 说明 |
|---|---:|---:|---|
| 重建项目原生库的本轮完整导出 | 19.14 s | 271.49 s | AOT 包含 197 单元的 C++ 编译和链接，约 14.19 倍 |
| 原生对象完全命中的无变更导出 | 7.07 s | 6.41 s | 两者接近，AOT 缓存命中后不再重复编译 |

因此首次导出或缓存失效时，C++ 工具链成本是当前最明显的劣势；无变化导出已能复用对象缓存。
后续基准需要继续拆分前端、代码生成、单对象编译、链接和场景转换耗时，并记录缓存命中率。

### 商业语料编译可扩展性

2026-07-16 使用当前编译器对三份商业语料执行了独立冷/热生成。Windows 11/MSVC 上 Castle
Defense（180 单元）为 8.09/0.30 秒，Dungeon Rush（197 单元）为 5.87/0.18 秒，NoahEngine
（122 单元）为 7.84/0.17 秒；热生成均为全量缓存命中。Castle 的
183 个原生目标冷链接为
87.04 秒、无变化热构建为 2.66 秒。macOS/AppleClang 对三者的生成分别为 0.49/0.09、
1.05/0.13、0.33/0.05 秒，完整原生链接分别为 37.29、35.87、26.67 秒。

Windows 数字包含 Defender、NTFS 和进程启动成本，不能与 macOS 直接比较编译器吞吐；它们的
价值是锁定同一主机上的冷/热比例和大翻译单元内存回归。37 个挂起点的 Castle 单元改为有界
MIR 状态机后从 368,708 B/1,416 行降至 156,157 B/692 行，MSVC 不再出现 `C1060`。根构建还
默认串行调度完整 godot-cpp SDK 变体、变体内部保持并行，避免版本数、profile 数与全核并行
相乘导致构建机内存和 Ninja 数据库争用。

本轮完整 AOT 导出曾在 `savepack` 后释放 `test_level.tscn` 的整棵导出临时场景时输出一次
`Can't emit non-existing signal "changed"`。导出器现在线程安全地记录 Godot 已排队删除的定制
场景，并在扩展仍完整注册时按子节点后序释放；使用完整场景转换缓存重建后重新导出，退出码为
0，编辑器和控制台日志中的 `ERROR`、`SCRIPT ERROR`、`WARNING` 均为 0。

随后故障注入发现 Godot 4.5 会把一次失败转换的原始 `.scn` 留在导出缓存中，而且 scene
customizer 的 configuration hash 没有正确进入缓存目录键。GDPP 现将转换修订和项目构建 ID
同时写入插件名称；修复后完整重转 317 个场景、替换 431 个脚本节点并复制 15,931 个存储属性，
不会复用旧缓存。`metadata/<key>` 通过 Object 元数据 API 单独复制，其他缺失属性在场景树变更
前就会原子失败。

运行期还发现生成的函数局部 `preload()` 缓存会晚于 Godot Rendering/Physics Server 销毁，
导致游戏强制退出时报告纹理、Shape 和 RID 残留。项目扩展现在在 scene 级 terminator 中按注册
逆序清空成员预加载、线程安全惰性 `const preload` 缓存和脚本 `static var` 存储，再让服务器退出。
Dungeon Rush 直接关卡运行 1800 帧后的退出日志中，资源泄漏、RID 残留、脚本错误和服务器错误
均为 0。

## 已落地的热路径与代码尺寸优化

- [x] 同步 `for i in range(...)` 直接生成整数循环，start/stop/step 只求值一次，不创建临时
  `Array` 或 `Variant` 迭代器，并检查零步长。
- [x] 十种 PackedArray 同步循环直接按 size/index 和静态元素类型读取，不经过 Variant 装箱。
- [x] 动态方法调用的 `StringName` 按调用点缓存，减少每帧重复哈希和分配。
- [x] 动态嵌套赋值把根对象、每层下标和右值各求值一次，再反向复用临时值回写；避免为了修复
  值类型副本语义而重复执行节点查找、属性 getter 或带副作用的键表达式。
- [x] Release 编译启用函数/数据 section，MSVC 使用 `/OPT:REF /OPT:ICF`，ELF/Mach-O 使用
  gc-sections/dead-strip。
- [x] 编辑器安全默认值只生成一次并保持 GDScript 声明顺序；不安全初始化仅增加运行期保护，
  不再复制整套初始化机器码。

这些优化已通过生成代码、原生链接和 Dungeon Rush 直接关卡 1800 帧回归。补充的三轮无上限
直接关卡短采样中，ori 得到 5 个每秒样本、中位数 475 FPS；AOT 得到 6 个样本、中位数
472.5 FPS，AOT 低约 0.5%，属于短样本运行波动，不能拿它宣称性能提高。项目 DLL 增大到约
4.01 MiB，说明 StringName 缓存、默认值保护和新增正确性代码也有尺寸成本，后续必须用固定
workload 决定哪些调用点值得缓存。

## 启动与游戏运行

| 指标 | ori | aot | 判断 |
|---|---:|---:|---|
| 主菜单就绪中位数 | 1031.76 ms | 1016.02 ms | AOT 快 15.74 ms（1.53%），幅度很小 |
| 菜单工作集均值 | 110.90 MiB | 111.24 MiB | AOT 多约 0.34 MiB |
| 60 FPS 垂直同步 | 60.0 ± 0.0 | 60.0 ± 0.0 | 无差异 |
| 垂直同步 CPU 核占用 | 29.87% | 29.13% | AOT 低约 2.47% |
| 垂直同步工作集 | 252.81 MiB | 247.30 MiB | AOT 低约 5.51 MiB |
| 解除上限平均 FPS | 238.37 | 239.18 | AOT 高 0.34%，处于运行波动范围 |
| 解除上限 FPS 标准差 | 1.71 | 1.78 | AOT 未改善稳定性 |
| 解除上限工作集 | 248.79 MiB | 243.38 MiB | AOT 低约 5.41 MiB |

本轮可确认的优势是源码不进入 PCK、运行工作集约低 2%、CPU 有小幅下降；启动差异很小，解除
上限后的平均 FPS 差异不足以证明有性能提升，帧率稳定性也没有改善。当前生成代码仍有较多
Variant 和 Godot 动态调用，且这组自动化主要覆盖 Home 场景，不是敌群、Boss、存档或长时间
压力测试。因此项目现阶段不能据此使用“超高性能”商业宣传。

另有一组专门用于生命周期回归的直接关卡测试：每个版本执行 7 次“进程启动、运行 2 帧并完整
退出”，ori 中位数为 3029.9 ms，AOT 为 2027.3 ms，AOT 缩短 1002.6 ms（33.1%）。该指标包含
启动、两帧执行和资源关闭，不等同于主菜单可交互时间；它说明此项目的脚本装载路径受益明显，
但不能替代上表更接近玩家体验的主菜单和 20 秒运行指标。

## 固定 GDScript/AOT 运行差分矩阵

2026-07-16 使用官方 Godot `4.5.2.stable.official.6ce3de25a` 在 macOS Apple Silicon、
Ubuntu 24.04 x86_64 和 Windows 11 x86_64 完成同一固定矩阵。GDScript 和 AOT 导出使用同一个
Godot 可执行文件；工具会读取两者 `--version`，不一致时直接判定 oracle 失败。每个平台执行
3 轮 AB/BA 交替启动、微基准和帧测试；每个微基准得到 15 个样本，每种实现得到 360 个计时帧。
完整机器可读报告和原始日志保存在根目录 `build/platform-matrix/`，该证据目录不进入插件交付包。

行为 oracle 比较数组和值类型嵌套写回、整数求和、lambda、`match`、字符串、信号结果，以及
`INT64_MAX + 1` 回绕和 64 位按位与等边界语义。全部轮次输出一致，日志中没有脚本错误、引擎
错误或运行时阈值失败。固定性能预算在三个系统上全部通过：

| 固定工作负载 | macOS AOT 加速 | Linux AOT 加速 | Windows AOT 加速 |
|---|---:|---:|---:|
| 强类型数值 | 9.93× | 19.95× | 13.79× |
| 强类型分支 | 22.05× | 74.67× | 85.08× |
| `Array` | 1.37× | 1.61× | 1.70× |
| PackedArray | 1.85× | 3.68× | 6.11× |
| `Dictionary` | 1.16× | 1.14× | 1.06× |
| 内建向量 | 12.08× | 38.19× | 64.03× |
| 对象属性 | 2.55× | 2.80× | 1.32× |
| 字符串操作 | 1.16× | 1.42× | 1.18× |
| 项目方法调用 | 72.76× | 128.99× | 70.99× |
| `Callable` | 1.62× | 1.80× | 1.55× |
| Variant 运算 | 1.45× | 1.98× | 1.84× |
| 信号发射 | 1.07× | 1.51× | 1.05× |
| 分配型集合构造 | 1.76× | 1.64× | 1.77× |

首轮 Windows 实测曾发现信号发射慢 43.81%，并被阈值正确阻断。根因是本地脚本信号每次发射
都构造临时 `godot::Signal`。代码生成器现对本地/`self` 信号直接调用 `emit_signal` 并缓存
`StringName`，外部接收者仍使用保持空值和错误语义的通用路径。修复后信号在三平台分别为
1.07×、1.51×、1.05×，没有放宽阈值。动态容器、字符串和信号的收益仍明显低于静态数值路径，
属于持续优化对象。

| 进程与发行指标 | macOS | Linux | Windows |
|---|---|---|---|
| 启动中位数 GDS/AOT | 160.814/166.336 ms | 164.360/166.905 ms | 157.444/157.467 ms |
| AOT 启动变化 | +3.434% | +1.548% | +0.015% |
| 有效 FPS GDS/AOT | 145.939/145.654 | 143.595/144.067 | 145.367/144.000 |
| 帧间隔 CV GDS/AOT | 0.03743/0.03758 | 0.01575/0.00887 | 0.01581/0.00000 |
| 固定帧工作量加速 | 11.75× | 23.19× | 24.11× |
| PCK GDS/AOT | 28,880/9,516 B | 28,880/9,420 B | 28,880/9,420 B |
| 完整发行物 AOT 变化 | +871,482 B（+0.464%） | +1,400,356 B（+1.995%） | +618,492 B（+0.642%） |
| RSS GDS/AOT | 未可靠采集 | 78.31/79.56 MB | 未可靠采集 |

当前固定示例的 AOT PCK 比 GDScript 小约 67.05%～67.38%，不再是旧报告中的 77.84%。原因是
当前 AOT PCK 正确包含项目运行描述符、原生场景和 Autoload 注册资源；旧样本较小且尚未覆盖
完整运行注册内容。这个比例只适用于 28 KB 的专用 fixture，不能推广到游戏。Dungeon Rush
的真实 64 MB PCK 在删除全部脚本后仍因 317 个场景的原生化序列化净增 0.285%。无论 fixture
的 PCK 是否变小，完整发行物都会增加项目动态库，三平台本轮分别增加 0.464%、1.995% 和
0.642%。

微基准刻意放大单一语言路径，不能外推成游戏整体帧率。macOS AOT 启动中位数高约 3.43%，
Windows 平均 FPS 低约 0.94%，Linux RSS 高约 1.60%；这些负面结果都保留在门禁
数据中。完整跨平台方法、环境和限制见[跨平台实测报告](PLATFORM_TEST_REPORT.md)。固定矩阵
仍需扩展到 4.4、4.6、4.7、场景生命周期、协程、资源图、长时间运行和手写 godot-cpp 对照，
才可用于“完整 GDScript 等价”或普遍性能宣传。
