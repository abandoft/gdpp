# GDPP 商业化交付路线图

本文档定义 GDPP 从可运行原型走向商业级 GDScript AOT 编译器的实施顺序、完成状态和发布
门槛。语言兼容范围以[语言兼容性](COMPATIBILITY.md)为准，组件边界以[架构](ARCHITECTURE.md)
为准，发行包要求以[商业交付模型](COMMERCIAL_DELIVERY.md)为准。

## 状态规则

`[x]` 表示代码已进入当前工程，并具有自动化测试或可重复的集成验证；`[ ]` 表示尚未实现、
只完成局部实现，或尚未达到商业发布门槛。部分完成的能力必须拆成独立条目，不能提前关闭
整个阶段。只有通过对应阶段的功能、兼容性、性能和交付门禁后，阶段才允许整体关闭。

状态只接受以下证据：可重复执行的单元/集成/端到端测试、受版本控制的构建或发布配置、
真实 Godot 运行结果、平台 runner 产物审计，以及与实现同步的用户文档。仅有设计意图、接口
占位、能够编译但没有行为测试、只在单一开发机手工成功，均保持 `[ ]`。

### 路线图治理

- [x] 所有可执行任务统一使用 `[x]` 或 `[ ]`，不使用“基本完成”“大概支持”等模糊状态。
- [x] 将语言兼容、架构、项目构建和商业交付文档作为路线图状态的约束来源。
- [x] 将未完全实现的复合能力拆分，避免一个子能力完成后关闭整组任务。
- [ ] 为每个合并请求关联路线图任务、测试证据、兼容性影响和缓存/ABI 影响。
- [ ] 在 CI 中检查路线图任务格式、失效的相对链接、重复任务和已删除功能的陈旧 `[x]`。
- [ ] 每个次版本冻结前执行一次状态审计，由实现、测试、文档和发行负责人共同确认。
- [ ] 为所有 `[x]` 能力建立可机器读取的 feature manifest，并从测试结果生成支持矩阵。

### 优先级与变更规则

- [x] P0 表示语义正确性、数据安全、构建事务性或导出安全问题，优先于新语法和性能优化。
- [x] P1 表示正式桌面交付、编辑器体验和可维护性问题，完成后才允许进入 Stable 候选。
- [x] P2 表示供应链、规模化发行和新增平台能力，不得绕过对应平台门禁直接声明支持。
- [ ] 为缺陷建立严重级别：阻断、严重、一般、轻微，并定义修复时限和发布阻断规则。
- [x] 任何改变生成 C++ 的公共头文件、前端、语义、IR、后端、项目编排、runtime、版本化 API
  表和生成工具都进入确定性代码生成契约指纹；manifest v3 自动拒绝旧指纹，不依赖人工递增。
- [ ] 任何扩展语言提案都必须证明不会无声明改变普通 `.gd` 的 Godot 兼容语义。

## 产品目标与不可破坏约束

- [x] 使用 C++17 实现独立编译器，不依赖 Godot 引擎源码内部模块。
- [x] 通过 godot-cpp 和 GDExtension 集成 Godot，最低支持 Godot 4.4。
- [x] 用户侧桌面构建只要求完整 C++ 工具链，不要求安装 CMake、Python、Ninja、SCons 或
  单独构建 godot-cpp。
- [x] `.gd` 兼容模式对未支持语法明确报错，不以近似翻译、静默忽略或复制源码代替正确实现。
- [x] 成功导出时只分发项目原生库和运行时描述符，不分发项目 GDScript、编译器插件或 SDK。
- [x] AOT 构建或资源替换失败时保留 GDScript；二进制预设默认阻断，只有显式兼容预设允许
  source fallback，不能静默交付源码。
- [ ] 建立正式的语言兼容承诺、ABI 兼容承诺、版本支持周期和弃用策略。
- [ ] 为增强语法建立显式、版本化、可迁移的选择机制，普通 `.gd` 始终保持 Godot 兼容语义。

## 交付阶段与依赖关系

```text
核心编译链：M0 -> M1 -> M2 -> M3 -> M4 -> M5 -> M6
平台与质量：M0 -> M7 -> M8 -> M9
产品化链路：M5 -> M10 -> M12 -> M13 -> M14
增强语言：M1 + M2 + M3 + Stable 核心门禁 -> M11
```

M7–M10 是横向门禁，不应等核心功能全部结束后才启动；M11 必须等待兼容核心稳定，避免用新
语法掩盖 GDScript 语义缺口。M12–M14 从 Preview 开始持续建设，并在 Stable 前成为阻断门禁。

- [ ] M4 后端/runtime 覆盖声明支持的 GDScript、Godot 生命周期和 ABI 安全边界。
- [ ] M5 项目编译、直接构建、缓存事务和编辑器体验达到生产可用。
- [ ] M6 场景/Resource 转换、二进制交付、fallback 和安全审计达到生产可用。
- [ ] M7 所有正式声明平台、架构和 Godot 版本通过真实运行矩阵。
- [ ] M8 正确性、fuzz、压力、差分和性能体系全部成为持续门禁。
- [ ] M9 CI/CD、供应链、签名、公证、符号和发布恢复流程全部完成。
- [ ] M10 用户文档、CLI、doctor、报告和 IDE 接入形成完整开发者体验。
- [ ] M11 GDPP 增强语言完成特性登记、成熟度分级和迁移机制后，才允许逐项进入实验状态。

- [ ] M13 结构化日志、trace、诊断包、崩溃索引和故障恢复完成。
- [ ] M14 离线、企业 CI、混合语言、第三方 addon 和制品仓库集成完成。

### 阶段 A — 内部工程闭环

- [x] 独立编译器能够完成词法、语法、语义、类型化 IR、C++ 生成和项目级批量编译。
- [x] compiler 插件、fallback、用户 SDK 和项目原生库具有清晰分工与独立产物名称。
- [x] `example` 项目能够通过直接 C++ 工具链构建，并在 Godot 中加载和执行生成类。
- [x] 4.4～4.7 API 目标、内容构建 ID、增量缓存和二进制导出主路径已接通。
- [ ] 清除所有会生成无效 C++ 的已知前端路径；未支持结构必须在代码生成前终止。
- [x] 建立固定提交的官方 GDScript 语料基线，量化单文件与项目上下文兼容率并持续构建四个
  官方项目的原生库。
- [ ] 完成阶段 A 退出审计；在差分运行和未支持语法前置拒绝完成前不关闭该阶段。

### 阶段 B — Developer Preview

- [ ] 完成 P0 语义正确性闭环、差分执行框架、fuzz 和 sanitizer 基线。
- [ ] macOS、Linux、Windows x86_64 均通过插件加载、项目构建、debug/release 导出和运行。
- [ ] 编辑器提供异步构建、取消、诊断跳转、工具链检查和可恢复失败状态。
- [ ] 发布明确的已支持语法清单、已知差异、升级方式和数据安全边界。
- [ ] Preview 包具有校验和、许可证清单、可复现归档布局和自动安装验证。

### 阶段 C — Public Preview / Beta

- [ ] 官方语料与真实样例项目达到批准的兼容覆盖率，阻断级差分问题清零。
- [ ] 增量构建、并发构建、热重载和重复导出通过长时间压力测试。
- [ ] 建立公开性能基线、回归预算、崩溃信息收集方式和兼容性问题模板。
- [ ] 完成复杂场景/Resource 引用图、Autoload、信号和生命周期行为验证。
- [ ] 冻结 1.0 的用户配置键、缓存格式、生成描述符和项目原生库 ABI 规则。

### 阶段 D — 1.0 Release Candidate

- [ ] 三桌面平台完成签名/公证、SBOM、来源证明、符号归档和最终包内容审计。
- [ ] RC 期间只接受阻断缺陷、文档修正和已批准的低风险兼容修复。
- [ ] 使用至少三个非示例 Godot 项目完成从安装、构建、导出、升级到回滚的验收。
- [ ] 完成安全审查、许可证审查、性能审查和灾难恢复演练。
- [ ] 所有 Stable 门禁拥有负责人、证据链接、批准记录和可重复验证命令。

### 阶段 E — Stable 与维护

- [ ] 发布 1.0 Stable，并执行 SemVer、支持矩阵、弃用周期和安全响应承诺。
- [ ] 建立补丁版本、紧急热修复、撤回发行、缓存迁移和客户通知流程。
- [ ] 连续两个次版本验证升级兼容性后，再评估 LTS 通道和长期 ABI 承诺。
- [ ] 移动端和 Web 只在各自完整构建/运行/导出门禁通过后进入正式支持矩阵。

## M0 — 工程基础与可复现构建

### 工程结构

- [x] 使用根 `CMakeLists.txt` 管理核心构建，不在 `src/` 中散布子级构建文件。
- [x] 按 `compiler`、`frontend`、`semantic`、`ir`、`codegen`、`project`、`runtime`、`godot`
  和 `cli` 划分源码职责。
- [x] 以镜像模块组织 `include/gdpp/<module>` 与 `src/<module>`，私有 Godot 适配代码保留在
  `src/integration/godot`，并由架构检查阻止平铺头文件和逆向依赖。
- [x] 将 godot-cpp 固定为 `third/godot-cpp` Git 子模块。
- [x] 禁止仓库根目录内的 in-source CMake 构建。
- [x] 将 GDPP 自身 CMake 中间产物限制在根 `build/` 下。
- [x] 将示例项目 AOT 工作区限制在 `example/addons/gdpp/build/`，最终动态库放入
  `example/addons/gdpp/binary/`。
- [x] development 动态库使用内容构建 ID，并在旧扩展卸载、新扩展加载成功后回收历史映像，
  防止 `binary/` 随脚本修改无限增长。
- [x] 使用 `.gitignore` 排除 CMake 产物、Godot 缓存、SDK、本机二进制和项目 AOT 产物。
- [x] 生成 `compile_commands.json`，为 clangd、clang-tidy 和静态分析提供统一编译数据库。
- [x] 从单一 CMake 项目版本生成编译器版本宏和 CLI `--version` 输出。
- [ ] 为缓存、manifest、SDK manifest、生成描述符分别定义独立格式版本和迁移策略。
- [ ] 建立仓库目录所有权说明，明确公共 API、私有实现、生成文件和发行资产的修改边界。
- [ ] 增加禁止提交 build 产物、SDK 二进制和 Godot 本地缓存的 CI 仓库卫生检查。

### 构建目标

- [x] 建立独立的 `gdpp_core` 编译器核心 target。
- [x] 建立 `gdpp` 命令行编译器 target。
- [x] 建立 `gdpp_godot_plugin` 编辑器编译服务 target。
- [x] 建立 `gdpp_runtime` 生成代码运行时 target。

- [x] 建立开发、插件和发行 CMake Preset。
- [ ] 增加安装规则和面向下游 CMake 项目的导出 target，仅在确有嵌入式编译器客户需求后启用。

### 基础质量

- [x] 开启严格编译警告，并支持将项目警告视为错误。
- [x] 建立 clang-format 规则和 CI 格式检查。
- [x] 建立 CTest 单元测试和 CLI 冒烟测试。
- [x] 建立 Godot GDExtension 加载测试。
- [ ] 在全部受支持编译器上启用一致的 warning baseline，维护平台特定豁免清单。

## M1 — 最新版 GDScript 前端兼容

### 前端架构稳定化

- [x] 确定官方语法策略：只维护当前声明支持的最新版稳定 GDScript，不为 Godot 4.4～4.7
  建立多套 lexer/parser 方言；旧写法是否兼容以最新版官方语法为准。
- [ ] 编译配置显式区分语言模式（GDScript compatible / GDPP enhanced）、GDPP feature set 与
  目标 Godot API；不增加官方语法版本选择器。
- [x] 将 AST 从通用 `kind + value + children` 迁移为 `std::variant` 强类型节点载荷和统一
  visitor；旧 kind/operand 方法仅保留为只读迁移适配器，使新增
  节点遗漏语义、lowering、优化或后端处理时能够在编译期或 verifier 门禁中暴露。
- [x] 建立 `LanguageFeatureRegistry` 基础，集中登记最新版官方注解的名称、合法目标、参数范围和
  行为类别，parser 与 semantic 不再维护互相漂移的注解白名单。
- [ ] 在 `LanguageFeatureRegistry` 上扩充 GDPP 增强特性的成熟度、启用方式、lowering、runtime
  shim、最低 GDPP/runtime ABI、诊断与测试，不用散落字符串条件实现扩展语言能力。
- [ ] 建立最新版语法基线更新清单；Godot 发布新的稳定版后，同步官方合法/非法语料、标准库
  能力和兼容差异，直接替换上一版基线而不是新增长期语法模式。

### 词法分析

- [x] 实现缩进、退格、换行、标识符、关键字、操作符和基本字面量扫描。
- [x] 为 Token 保存源文件、行、列和范围信息。
- [x] 对非法字符、错误缩进和未闭合结构生成结构化诊断。
- [x] 在括号、方括号和花括号内抑制结构换行，使跨行表达式保持单一语法单元。
- [x] 支持最新版十进制、二进制、十六进制、科学计数法、前导/尾随小数点和合法下划线位置；
  `0o` 八进制不是当前 GDScript 语法，作为非法输入明确拒绝。
- [x] 统一规范化整数基数与有符号 64 位边界；浮点按 Godot `built_in_strtod` 的 18 位尾数、
  10 次幂和指数钳制规则计算，再输出可跨 AppleClang/GCC/MSVC 精确复现的 C++ 常量，覆盖
  最大有限值、INF、NAN、下溢和 `0e400` 等宿主 `strtod` 不等价边界。
- [x] 实现单/双引号、raw、三引号、raw 三引号、反斜杠续行、全部文档化控制转义、UTF-16/
  UTF-32 转义与代理对；非法转义失败关闭，NUL 按 Godot 行为替换为 U+FFFD。
- [x] 普通、raw、三引号字符串及数字极值已贯通 lexer、强类型 AST/HIR、C++17 后端、真实
  Godot 4.7 GDS oracle 与原生 GDExtension 运行 oracle。
- [ ] 补齐最新版注释边界语料，并把所有非法字面量的诊断范围逐字节校准到 Godot。
- [x] 固定 Godot 4.7 stable 使用的 Unicode 17.0 `XID_Start`/`XID_Continue` 范围；UTF-8
  失败关闭且不做 Unicode 规范化，组合/预组字符保持不同符号，源码列按码点、范围按字节记录。
- [x] 验证 CRLF、LF、文件末尾无换行、UTF-8 BOM、Tab/空格混用、全文件缩进字符一致性、
  空白行和注释行；结构缩进与续行/括号内装饰缩进分开处理。
- [x] 将非 ASCII、C++ 关键字和实现保留名称编码为确定性的纯 ASCII 原生符号；完整 UTF-8
  字节编码与保留前缀转义保证映射单射，避免不同 GDS 标识符在 C++ 后端碰撞。
- [x] 通过 `CompileOptions::frontend_limits` 集中限制源码字节、单行字节、Token 数量、字面量
  字节、缩进层数、分组层数、parser 递归和详细诊断数量；任一边界超限均以稳定错误码失败关闭，
  强制补齐 EOF，并且不会进入语义分析或代码生成。
- [ ] 建立从 Godot 官方测试语料提取的 lexer 回归集。
- [ ] 对 lexer 建立持续 coverage-guided fuzz；当前已有固定种子 512 组恶意随机字节，以及
  Unicode、非法 UTF-8/NUL、极端缩进、超长源码/行/字面量和 Token 洪泛的常驻回归。

### 语法分析

- [x] 支持 `extends`、`class_name`、字段、常量、函数、信号、枚举和静态函数声明。
- [x] 支持类型标注、默认参数、返回类型、类型化数组和推断声明 `:=`。
- [x] 支持数组、字典、成员访问、下标、调用以及常用单目和双目表达式。
- [x] 支持 `is` 对 Godot 对象、项目脚本类型和内建值类型进行类型测试；动态对象走
  `Object::cast_to`，动态值类型比较 `Variant::Type`。
- [x] 支持 `if/elif/else`、`while`、`for`、`return`、`pass`、`break` 和 `continue`。
- [x] 支持常量模式、多备选模式、通配符、变量绑定和 `when` 守卫形式的 `match`。
- [x] 支持 `assert(condition, message?)`。
- [x] 支持 Godot 4 内联属性 `get`/`set(value)`。
- [x] 支持 `set = method` / `get = method` 访问器绑定，验证绑定方法签名并阻止属性自递归。
- [x] 支持当前兼容表列出的常用 `@export` Inspector 注解。
- [x] 支持幂、整数除法、位运算、移位和对应复合赋值的当前类型矩阵。
- [x] 支持 `is not`，并按 GDScript 规则处理 `not` 与比较/类型测试的优先级。

## M11 — GDPP 增强语言

增强能力不能破坏普通 `.gd` 的 Godot 兼容性。每项扩展必须先形成设计文档，明确解析方式、
类型规则、运行时语义、调试体验、成熟度、最低 GDPP/runtime ABI 和退化/迁移方案。官方语法
始终跟进最新版，不为增强语言复制官方语法版本；版本管理只约束 GDPP 特性和运行时契约。

### 扩展机制

- [ ] 确定增强模式的显式启用方式，例如独立扩展名、项目清单或文件级 feature 声明。
- [ ] 建立扩展特性 ID、实验/预览/稳定/弃用状态、能力协商和最低 GDPP/runtime ABI 检查。
- [ ] 为增强源码提供转换为兼容 GDScript 子集的可行性评估。
- [ ] 禁止增强模式无声明地改变现有 GDScript 表达式或控制流语义。
- [ ] 扩展文件与普通 `.gd` 具有明确互操作边界，普通 Godot 编辑器不会误加载不兼容源码。
- [ ] 扩展能力支持 feature flag、最低编译器/runtime 版本、弃用警告和自动迁移检查。
- [ ] 为扩展规范建立提案模板、评审期、实验期、稳定条件和撤回条件。
- [ ] 编译期执行必须可复现、可限制资源、默认无网络/文件系统副作用并可审计。

### Python 风格能力候选

- [ ] 设计模块、包、显式导入和可见性规则。
- [ ] 设计迭代器协议、生成器和 `yield` 状态机。
- [ ] 设计装饰器/属性宏，并限制其编译期副作用。
- [ ] 设计列表、字典和集合推导式。
- [ ] 设计结构化解构、模式绑定和更完整的 `match`。
- [ ] 设计上下文管理或确定性资源释放机制，不直接复制与 Godot 生命周期冲突的 Python 语义。
- [ ] 扩充标准库，同时明确哪些 API 由 Godot、GDPP runtime 或纯编译期实现提供。
- [ ] 为每项扩展提供格式化、LSP、调试、文档和迁移支持后再标记稳定。
- [ ] 设计异常/错误处理时优先兼容 Godot `Error`、错误宏和对象生命周期，不照搬 Python 异常。
